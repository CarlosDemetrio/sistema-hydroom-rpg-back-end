## CRITICIDADE A — BLOQUEADORES DE CAMINHO CRÍTICO

### GAP-01: Wizard de Criação de Ficha — Estrutura e Contrato de API Divergentes

**O que está indefinido**

O `FichaFormComponent` tem 10 seções em scroll único mas o submit atual envia apenas `{ nome }`. O contrato correto do `CreateFichaRequest` é `nome, jogadorId, racaId, classeId, generoId, indoleId, presencaId, isNpc`. Raça, classe, gênero, índole e presença não são persistidos — todos os cálculos derivados (vida, bônus de classe) estão errados.

**Opções:**
- A: Wizard de 3 passos (Básico → Identidade → Confirmação) — alinhado ao contrato atual
- B: Wizard de 5-6 passos incluindo atributos e descrição física — experiência completa
- C: Formulário mínimo (nome + identidade) + completar depois na tela de detalhe

resposta: opcao B, lembre-se de incluir um auto save enquanto o usuário preenche o wizard, para evitar perda de dados, salve no backend a versao incompleta, mas só permita entrar na sessao com ela completa

**Perguntas para o PO:**
1. O que é obrigatório na criação? Apenas nome, ou também raça e classe?
todos campos são necessários serem preenchidos para criar a ficha, mas o auto save deve salvar a versão incompleta, para evitar perda de dados
2. Atributos são distribuídos na criação ou depois, na tela de detalhe?
atributos são preenchidos na criação. depois é a edicao conforme o usuário recebe EXP e sobe de nível
3. Insólitus, Título Heróico e Arquétipo entram no wizard ou só na edição?
Insólitus é algo que o mestre que pode conceder, pode acontecer dele conceder no inicio do jogo, mas não é obrigatório para criar a ficha, então pode ser algo que o mestre conceda depois, na edição da ficha.
Título Heróico e Arquétipo são coisas que o mestre concede

### GAP-02: Sistema de XP — Quem Concede, Quando e Como

**O que está indefinido**

`PUT /fichas/{id}` aceita campo `xp` com `hasAnyRole('MESTRE', 'JOGADOR')` — jogador pode alterar a própria XP. Sem endpoint de concessão em lote. Sem histórico de concessão.

**Perguntas para o PO:**
1. Jogador pode editar a própria XP ou é campo read-only para jogador?
read only para o jogador, apenas o mestre pode conceder XP, o jogador só pode ver a XP atual, mas não editar
2. Concessão em lote (todas as fichas do jogo) é requisito para o modo sessão?
não é requisito, mas seria uma funcionalidade interessante para o mestre, para conceder XP para todos os jogadores de uma vez
3. Histórico de concessão de XP é requisito agora ou futuramente?
histórico é algo que pode ser implementado futuramente, não é requisito para o MVP, mas seria interessante para o mestre ter um registro de quando concedeu XP

### GAP-03: VantagemEfeito — Backend Implementado, Motor Desconectado

**O que está indefinido**

8 tipos de `VantagemEfeito` implementados no backend com CRUD completo, mas `FichaCalculationService` não os consome. Campo `VT` hardcoded. Fichas com vantagens de bônus têm valores matematicamente errados.

**Perguntas para o PO:**
1. Integração de VantagemEfeito no motor de cálculo é prioridade antes do FichaDetail ser "pronto"?
sim, TODOS as configuracoes têm de estar 100% funcionais antes de comecar a se preocucar com modulo de ficha.
2. `DADO_UP`: onde o valor do dado atual é armazenado/exibido na ficha?
o dado é para setar apenas o dado que será jogado, ele não tem um valor atual. o jogador pode escolher jogar com um dado de 6, 8, 10, 12 ou 20 faces,
3. `FORMULA_CUSTOMIZADA`: quais variáveis estão disponíveis (nivel_vantagem, atributos, nivel_personagem)?
todas que fizerem sentido para o contexto

## CRITICIDADE B — ALTA PRIORIDADE

### GAP-05: NPC vs Ficha de Jogador — Diferenças de Negócio

**Perguntas para o PO:**
1. NPC é mecanicamente idêntico à ficha de jogador (mesmo motor de cálculo)?
sim, NPC é uma ficha como outra qualquer, a unica diferenca é que ela tem o campo isNpc=true, e o mestre pode escolher se quer mostrar ou nao as estatisticas dela para os jogadores
2. Campo `descricao` existe apenas em NPC ou também em fichas de jogadores?
para jogadores também, mas é um campo opcional
3. Mestre pode revelar stats de NPC para jogadores? (`visivelParaJogadores` booleano)
pode, e tem que ser possivel mostrar a apenas um jogador, pode ocorrer o caso que este npc é uma invocacao, ou besta parceira do jogador
### GAP-06: Nível — Cálculo de Pontos Disponíveis

`FichaResumoResponse` não inclui `pontosAtributoDisponiveis`, `pontosAptidaoDisponiveis`, `pontosVantagemDisponiveis`. Frontend usa `0` hardcoded.

**Perguntas para o PO:**
1. Pontos acumulam ao longo dos níveis (saldo = ganhos - gastos) ou apenas os do nível atual são distribuíveis?
podem acumular, mas se gastos, não pode ser reversivel
2. Level up: automático ao acumular XP ou wizard de confirmação obrigatório?
automático. ele apenas libera os valores que o jogador pode distribuir, mas não distribui automaticamente, o jogador tem que escolher onde gastar os pontos
3. `FichaResumoResponse` precisa incluir os três campos de pontos disponíveis?
se a tela precisar, sim
### GAP-07: Essência — Persistência de Estado de Gasto

`FichaResumoResponse` tem `essenciaTotal` mas não `essenciaAtual`. Barra de essência mostra 100% sempre (hardcoded).

**Perguntas para o PO:**
1. `essenciaGasta` persiste entre sessões ou reseta ao "descansar"? Quem controla?
2. Endpoint `PUT /fichas/{id}/essencia` deve existir simetricamente ao `PUT /fichas/{id}/vida`?
o mestre pode resetar todos os estados, incluindo vida, essencia e etc.
### GAP-08: Prospecção — Semântica do Endpoint

US-046: Mestre adiciona (read-only para jogador). US-065: Jogador usa no modo sessão. Dois atores, operações opostas, endpoint genérico único.

**Perguntas para o PO:**
1. Dois endpoints separados: `POST /fichas/{id}/prospeccao/conceder` (MESTRE) e `POST /fichas/{id}/prospeccao/usar` (JOGADOR)?
2. Usar prospecção precisa de confirmação (é irreversível)?
o jogador usa e o mestre marca que ele usou, mas o mestre tem a opção de reverter caso queira, mas o jogador não pode reverter sozinho

## CRITICIDADE C — MÉDIA PRIORIDADE

### GAP-09: Insólitus — Narrativa Livre vs Modelagem Mecânica
Glossário define impacto mecânico real, mas modelo atual é apenas `varchar`. Necessita decisão antes dos campos de criação da ficha.
insólitus, entra na mesma categoria das vantagens, ela pode permitir praticamente qualquer coisa, uma raca, uma classe, uma vantagem, bonus, aptidao e etc
### GAP-10: Onboarding — Role não é Selecionável
US-004 descreve wizard de escolha de role, mas role é definida no OAuth2 e não pode ser alterada. Role é global — não suporta "Mestre em um jogo, Jogador em outro".
isso é uma falha de modelagem do sistema, deveria haver um perfil admin, um de mestre e outro de jogador. o perfil admin teria acesso a tudo, o de mestre teria acesso a tudo relacionado a jogos que ele criou, e o de jogador teria acesso a tudo relacionado a jogos que ele participa como jogador. isso permitiria uma flexibilidade muito maior, e resolveria esse gap de onboarding.
### GAP-11: Histórico — Visibilidade por Role
Envers configurado mas regras de visibilidade não definidas. Quando Mestre edita ficha de jogador, o jogador vê o evento?
envers é bobeira para nosso MVP, nao precisamos disso no momento.
### GAP-12: Duplicação — Semântica do `manterJogador`
Se `manterJogador=true`, cria dois personagens para o mesmo jogador no mesmo jogo. Permitido? Mestre pode converter NPC em ficha de jogador via duplicação?
pode ocorrer de uma ficha de um jogador morrer e ele ter que criar outra, inclusive isso ocorre com bastante frequencia
